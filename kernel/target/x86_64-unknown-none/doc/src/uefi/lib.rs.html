3//! Rusty wrapper for the [Unified Extensible Firmware Interface][UEFI].
4//!
5//! This crate makes it easy to develop Rust software that leverages **safe**,
6//! **convenient**, and **performant** abstractions for [UEFI] functionality.
7//!
8//! See the [Rust UEFI Book] for a tutorial, how-tos, and overviews of some
9//! important UEFI concepts. For more details of UEFI, see the latest [UEFI
10//! Specification][spec].
11//!
12//! # Minimal Example
13//!
14//! Minimal example for an UEFI application using functionality of the
15//! `uefi` crate:
16//!
17//! ```no_run
18//! #![no_main]
19//! #![no_std]
20//!
21//! use uefi::prelude::*;
22//!
23//! #[entry]
24//! fn main() -> Status {
25//!     uefi::helpers::init().unwrap();
26//!
27//!     Status::SUCCESS
28//! }
29//! # extern crate std;
30//! ```
31//!
32//! Please find more info in our [Rust UEFI Book].
33//!
34//! # Value-add and Use Cases
35//!
36//! `uefi` supports writing code for both pre- and post-exit boot services
37//! epochs, but its true strength shines when you create UEFI images that heavily
38//! interact with UEFI boot services. Still, you have the flexibility to just
39//! integrate selected types and abstractions into your project, for example to
40//! parse the UEFI memory map.
41//!
42//! _Note that for producing UEFI images, you also need to use a corresponding
43//! `uefi` compiler target of Rust, such as `x86_64-unknown-uefi`._
44//!
45//! ## Example Use Cases
46//!
47//! This library significantly simplifies the process of creating **UEFI images**
48//! by abstracting away much of the UEFI API complexity and by providing
49//! convenient wrappers. When we mention UEFI images, we are talking about UEFI
50//! applications, UEFI boot service drivers, and EFI runtime service drivers,
51//! which typically have the `.efi` file extension. For instance, an UEFI
52//! application could be an OS-specific loader, similar to _GRUB_ or _Limine_.
53//!
54//! Additionally, you can use this crate in non-UEFI images (such as a kernel
55//! in ELF format) to perform tasks like parsing the UEFI memory map embedded in
56//! the boot information provided by a bootloader. It also enables access to
57//! UEFI runtime services from a non-UEFI image kernel.
58//!
59//! # Supported Compiler Versions and Architectures
60//!
61//! `uefi` works with stable Rust, but additional nightly-only features are
62//! gated behind the `unstable` Cargo feature. Please find more information
63//! about additional crate features below.
64//!
65//! `uefi` is compatible with all platforms that both the Rust compiler and
66//! UEFI support, such as `i686`, `x86_64`, and `aarch64`. Please note that we
67//! can't test all possible hardware/firmware/platform combinations in CI.
68//!
69//! ## MSRV
70//!
71//! The minimum supported Rust version is currently 1.81.
72//! Our policy is to support at least the past two stable releases.
73//!
74//! # API/User Documentation, Documentation Structure, and other Resources
75//!
76//! Down below, you find typical technical documentation of all types, modules,
77//! and functions exported by `uefi`.
78//!
79//! For a TL;DR quick start with an example on how to create your own EFI
80//! application, please check out [the UEFI application template][template]. The
81//! [Rust UEFI Book] is a more beginner-friendly tutorial with How-Tos, and
82//! overviews of some important UEFI concepts and the abstractions provided by
83//! this library.
84//!
85//! For more details of UEFI itself, see the latest [UEFI Specification][spec].

87//! # Library Structure & Tips
88//!
89//! The top-level module contains some of the most used types and macros,
90//! including the [`Handle`] and [`Result`] types, the [`CStr16`] and
91//! [`CString16`] types for working with UCS-2 strings, and the [`entry`] and
92//! [`guid`] macros.
93//!
94//! ## UEFI Strings
95//!
96//! Rust string literals are UTF-8 encoded and thus, not compatible with most
97//! UEFI interfaces. We provide [`CStr16`] and [`CString16`] for proper working
98//! with UCS-2 strings, including various transformation functions from standard
99//! Rust strings. You can use [`ctr16!`] to create UCS-2 string literals at
100//! compile time.
101//!
102//! ## Tables
103//!
104//! Most UEFI functionality comes from the system, boot, and runtime
105//! tables. These can be accessed via the [`system`], [`boot`], and [`runtime`]
106//! modules.
107//!
108//! ## Protocols
109//!
110//! When boot services are active, most functionality is provided via UEFI
111//! protocols. Protocols provide operations such as reading and writing files,
112//! drawing to the screen, sending and receiving network requests, and much
113//! more. The list of protocols that are actually available when running an
114//! application depends on the device. For example, a PC with no network card
115//! may not provide network protocols.
116//!
117//! See the [`boot`] documentation for details of how to open a
118//! protocol, and see the [`proto`] module for protocol implementations. New
119//! protocols can be defined with the [`unsafe_protocol`] macro.
120//!
121//! ## Optional Cargo crate features
122//!
123//! - `alloc`: Enable functionality requiring the [`alloc`] crate from
124//!   the Rust standard library. For example, methods that return a
125//!   `Vec` rather than filling a statically-sized array. This requires
126//!   a global allocator; you can use the `global_allocator` feature or
127//!   provide your own. This is independent of internal direct usages of the
128//!   UEFI boot service allocator which may happen anyway, where necessary.
129//! - `global_allocator`: Set [`allocator::Allocator`] as the global Rust
130//!   allocator. This is a simple allocator that relies on the UEFI pool
131//!   allocator. You can choose to provide your own allocator instead of
132//!   using this feature, or no allocator at all if you don't need to
133//!   dynamically allocate any memory.
134//! - `logger`: Logging implementation for the standard [`log`] crate
135//!   that prints output to the UEFI console. No buffering is done; this
136//!   is not a high-performance logger.
137//! - `panic_handler`: Add a default panic handler that logs to `stdout`.
138//! - `unstable`: Enable functionality that depends on [unstable
139//!   features] in the nightly compiler.
140//!   As example, in conjunction with the `alloc`-feature, this gate allows
141//!   the `allocator_api` on certain functions.
142//! - `qemu`: Enable some code paths to adapt their execution when executed
143//!   in QEMU, such as using the special `qemu-exit` device when the panic
144//!   handler is called.
145//!
146//! Some of these features, such as the `logger` or `panic_handler` features,
147//! only unfold their potential when you invoke `uefi::helpers::init` as soon
148//! as possible in your application.

159//! # Comparison to other Projects in the Ecosystem
160//!
161//! ## Rust `std` implementation
162//!
163//! There is an ongoing effort for a [`std` implementation][rustc-uefi-std] of
164//! the Rust standard library, which allows you to write UEFI programs that look
165//! very similar to normal Rust programs running on top of an OS.
166//!
167//! It is still under development. You can track the progress in the
168//! corresponding [tracking issue][uefi-std-tr-issue].
169//!
170//! Using the `std` implementation simplifies the overall process of producing
171//! the binary. For example, our [`#[entry]`][entry-macro] macro won't be
172//! required any longer. As the `std` implementation evolves over time, you'll
173//! need fewer and fewer abstractions of this crate. For everything not covered
174//! by the `std` implementation, you can obtain relevant structures to work with
175//! our crate via:
176//! - `std::os::uefi::env::boot_services()`
177//! - `std::os::uefi::env::get_system_handle()`
178//! - `std::os::uefi::env::get_system_table()`
179//!
180//! ## `r-efi`
181//!
182//! [`r-efi`] provides Raw UEFI bindings without high-level convenience similar
183//! to our `uefi-raw` crate, which is part of this  project, but more
184//! feature-complete. It targets a lower-level than our `uefi` crate does.
